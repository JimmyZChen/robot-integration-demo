## ğŸ§© Key Design: RabbitMQ Asynchrony & Peak-Shaving
- **Design**: integrate RabbitMQ and build an async task channel (Topic â†’ Queue â†’ DLQ) with **manual ack** and **idempotency checks**.  
- **Contract**: APIs return **202 Accepted** with a `taskId`; the client polls for status. Burst traffic is smoothed.  
- **Config**: MQ parameters are delivered via Nacos; failures can be quickly traced in the DLQ.

**Acceptance â†’ Query (example)**
```http
POST /external/gs/async/robot/command/tempTask  â†’ 202 Accepted {taskId}
GET  /external/gs/async/tasks/{taskId}          â†’ PENDING | DONE | FAILED
```

**Topology (consistent naming for easy console verification)**
- `Exchange=robot.task.topic`ï¼ˆtopicï¼‰ â†’ `Queue=robot.task.q`
- `DLX=robot.task.dlx` â†’ `DLQ=robot.task.dlq`ï¼ˆ`rk=#`ï¼‰
- `RoutingKey=robot.task.dispatch`

**Consumer policy**
- Manual ack; on exception nack(requeue=false) â†’ goes directly to **DLQ**
- Reference settings: concurrency=2, prefetch=20 (up to 40 in-flight)
- Producer confirm/return + mandatory=true (routing failures observable)

**Idempotency & result cache**
- Idempotency key: request header X-Request-Id â†’ robot:idem:{id} (default TTL 3600s)
- Task status: robot:task:{taskId} (default TTL 86400s) for the query API

**Message model (sample)**
```json

  "taskId": "c34c7980...f1b7",
  "requestId": "rq-0002",
  "type": "GS_TEMP_TASK",
```
